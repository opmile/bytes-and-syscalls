# RTT — o essencial

**O que é**

RTT (round-trip time) é o tempo total entre disparar uma requisição e receber a resposta (em ms). 

* Não é igual a latência: latência é o trajeto de ida (one-way)

* RTT = ida + processamento no servidor + volta. Por isso RTT inclui o tempo de processamento do servidor remoto, não só rede.

**Faixas de referência (AWS)**

- < 100ms: bom, performance ótima
- 100–200ms: degradação perceptível, mas usável
- 200–375ms: experiência ruim, esperas longas
- \> 375ms: conexões tendem a cair

**O que afeta RTT**

1. Distância física (luz tem limite — servidor longe = RTT maior, sempre)

2. Congestionamento de rede (mais nós, mais fila)

3. Tempo de resposta do servidor (DB lento, APIs externas, fila de requests)

4. A própria rede interna da empresa (LAN pode ser gargalo mesmo com internet boa).

**Como pensar isso em sistema**

O insight central: **RTT é um orçamento que você gasta a cada interação do usuário**. Cada chamada serial (request → DB → API externa → response) soma RTTs. Por isso decisões de arquitetura como:

- *Paralelizar chamadas* em vez de encadear reduz RTT acumulado.
- *Cache próximo do usuário* (CDN, edge) ataca a parte "distância física" que você não pode otimizar via código.
- *Reduzir round-trips* (batch de queries, GraphQL agregando, HTTP/2 multiplexando, keep-alive) é frequentemente mais impactante que otimizar lógica do servidor.
- *Localização do deploy* importa: região AWS perto do usuário não é detalhe de infra, é decisão de UX.

A regra prática útil: ao desenhar um fluxo, conte quantos round-trips ele exige. Cada um custa pelo menos o RTT base entre os dois pontos — e esse custo é piso, não teto. É o tipo de coisa que não aparece em testes locais e explode em produção.

# Exemplo: paginação de feed

Imagine que você está construindo um endpoint `/feed` que retorna posts para o usuário. Cada post tem autor, contador de likes, e se o usuário logado curtiu ou não.

**Abordagem ingênua (N+1):**

```
GET /feed
  → SELECT * FROM posts LIMIT 20         (1 query)
  → para cada post: SELECT autor          (20 queries)
  → para cada post: SELECT count(likes)   (20 queries)
  → para cada post: SELECT liked_by_me    (20 queries)
```

61 round-trips ao banco. Se o RTT app↔DB é 2ms (mesma região AWS), são ~120ms só em rede, antes de qualquer query rodar. Se app e DB estão em regiões diferentes (digamos 40ms RTT), vira 2.4 segundos. O endpoint trava.

A explosão vem do loop. Você faz 1 query para pegar 20 posts, e aí itera sobre eles fazendo queries adicionais por post:

```
1  query: SELECT * FROM posts LIMIT 20

para cada um dos 20 posts:
  +1  query: SELECT autor WHERE id = post.autor_id
  +1  query: SELECT count(*) FROM likes WHERE post_id = post.id
  +1  query: SELECT EXISTS(...) WHERE user_id = me AND post_id = post.id
```

São 3 queries por post × 20 posts = 60. Mais a query inicial = 61.

É o clássico problema **N+1**: 1 query inicial + N queries derivadas (uma por item do resultado). Acontece naturalmente quando você escreve código tipo `posts.map(p => fetchAuthor(p.authorId))` — cada iteração do loop é um round-trip separado ao banco.

A correção é coletar os IDs e fazer **uma** query com `WHERE id IN (...)`, transformando N queries em 1.

**Abordagem agregada:**

```
GET /feed
  → 1 query com JOIN ou 3 queries com WHERE id IN (...)
```

3 round-trips, ~6ms de rede. A query em si pode ser mais pesada para o DB processar, mas o ganho de eliminar dezenas de round-trips domina.

A ideia é: em vez de perguntar ao banco 20 vezes "qual o autor desse post?", você pergunta uma vez só "quais são os autores desses 20 posts?".

**Passo 1 — pegar os posts:**

```sql
SELECT id, autor_id, conteudo FROM posts LIMIT 20;
```

Isso te devolve 20 linhas. No código, você extrai os `autor_id` num array: `[7, 12, 7, 3, 19, ...]`.

**Passo 2 — pegar todos os autores de uma vez:**

```sql
SELECT id, nome, avatar FROM usuarios WHERE id IN (7, 12, 3, 19, ...);
```

Uma query, um round-trip, traz todos os autores únicos. No código, você monta um mapa `{ 7: {...}, 12: {...} }` e faz o "join" em memória ao montar a resposta.

**Passo 3 — likes em batch também:**

```sql
SELECT post_id, COUNT(*) as total
FROM likes
WHERE post_id IN (1, 2, 3, ..., 20)
GROUP BY post_id;
```

Uma query devolve a contagem de likes de todos os 20 posts agrupados.

**Resultado:** 3 queries totais (posts, autores, likes) em vez de 61. O banco faz mais ou menos o mesmo trabalho — talvez até menos, porque consegue otimizar uma query agregada melhor que 60 queries pontuais — mas você só paga RTT 3 vezes.

A alternativa é um `JOIN` único:

```sql
SELECT p.id, p.conteudo, u.nome, COUNT(l.id) as likes
FROM posts p
JOIN usuarios u ON u.id = p.autor_id
LEFT JOIN likes l ON l.post_id = p.id
GROUP BY p.id, u.nome
LIMIT 20;
```

1 round-trip só. Trade-off: query mais complexa para o banco processar, mas elimina o trabalho de juntar os dados em memória no app.

**O trade-off real**

Agora chega o pedido: "queremos mostrar comentários recentes em cada post também". Você tem duas escolhas:

1. **Embutir tudo na resposta do feed.** Mais um JOIN, payload maior, query mais complexa. Você economiza um round-trip do cliente, mas paga em complexidade de query e bytes trafegados — mesmo para usuários que nunca expandem comentários.

2. **Deixar o cliente buscar comentários sob demanda** (`GET /posts/:id/comments` quando o usuário clica). Feed permanece leve, mas cada expansão custa um RTT cliente↔servidor (digamos 80ms numa rede 4G).

A escolha depende de telemetria: se 90% dos usuários expandem comentários, agregar no feed ganha. Se 10%, você está pagando RTT desnecessário em queries pesadas para a maioria.

**O ponto que importa**

RTT te força a escolher *onde* gastar o orçamento. Você pode:

- Gastar no servidor (queries mais complexas, payloads maiores) para economizar RTTs no cliente
- Gastar no cliente (mais requests) para manter o servidor simples e responsivo
- Gastar em infra (cache, CDN, replicação) para não gastar em nenhum dos dois

Não existe escolha "grátis". E é por isso que decisões como REST vs GraphQL, SSR vs CSR, monolito vs microsserviços não são religião — são alocação de orçamento de RTT contra outras restrições (complexidade, custo, manutenibilidade). Microsserviços, por exemplo, transformam o que era uma chamada de função em memória num round-trip de rede. Vale quando o ganho organizacional/escala compensa esse custo. Não vale quando você só tem 3 devs e todo request precisa atravessar 5 serviços para responder.
